# Homepage 대시보드 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** k3s 홈랩에 운영자 단일 진입점 대시보드 Homepage(gethomepage.dev)를 platform 컴포넌트로 추가해, 모든 서비스를 한 화면(dash.home.ukyi.app)에서 타일+가벼운 상태+인프라 요약으로 본다.

**Architecture:** adguard 동형의 plain manifest + kustomize 컴포넌트(`platform/homepage/prod/`). ArgoCD `platform-components` ApplicationSet이 자동 발견. 서비스는 HTTPRoute `gethomepage.dev/*` annotation으로 자동 카탈로그(`gateway: true`). 인프라 요약은 metrics-server 대신 VictoriaMetrics PromQL(`prometheusmetric` 위젯)로. 전용 `homepage` ns(PSA restricted) + read-only ClusterRole + 신규 NetworkPolicy.

**Tech Stack:** Kubernetes(k3s), Gateway API(HTTPRoute), ArgoCD, kustomize, gethomepage/homepage v1.13.x, VictoriaMetrics, bats(렌더 단언), kubeconform, conftest(원장).

**설계 문서:** `docs/plans/2026-06-18-homepage-design.md` (승인 완료).

- **작업 위치**: 이미 worktree `.claude/worktrees/feat+homepage`(브랜치 `worktree-feat+homepage`). 여기서만 작업.
- **테스트 방식**: 이 작업은 k8s manifest다. "TDD"는 **렌더 단언 테스트(bats + grep)를 먼저 쓰고 실패시킨 뒤 manifest를 작성**하는 형태다(pytest 아님). adguard의 `test_*.bats`가 그 패턴의 원형이다.

- **커밋 규약**: 한국어 conventional(`feat:`/`fix:`/`docs:`/`test:`/`chore:`), AI 마커 금지. 각 Task 끝 `Commit` 단계에서 현재 worktree 브랜치에 직접 커밋(`Skill(commit)` 호출 금지 — Phase D 실행 지침 따름).

## Task 1: `homepage` 네임스페이스 추가 (PSA restricted)

**Files:**
- Modify: `platform/namespaces/prod/namespaces.yaml`
- Modify: `platform/namespaces/prod/test_psa.bats` (기존 카운트 게이트 5→6 — F1)
- Test: `platform/namespaces/prod/test_homepage_ns.bats`

**근거:** AGENTS.md — appset 컴포넌트의 대상 ns는 `platform/namespaces`가 소유한다(appset에 destination.namespace 없어 CreateNamespace는 PSA 라벨 없는 ns만 만든다). Homepage는 restricted 호환(runAsUser 1000·privesc off·drop ALL·seccomp).

**(F4/F6) ArgoCD sync 순서 — 신규 ns의 실제 race:** namespaces-prod와 homepage-prod는 독립 Application이라 cross-Application 순서가 없고, **이 레포 appset은 `destination.namespace`가 없어 `CreateNamespace=true`가 homepage ns를 만들지 못한다**(AGENTS.md 명시). 따라서 homepage-prod가 homepage ns 생성 전에 reconcile하면 리소스 적용이 실패하고, ArgoCD sync op의 retry가 소진되면 수동 resync가 필요할 수 있다. 기존 컴포넌트(adguard 등)는 ns가 이미 존재해 안 겪지만 homepage는 **신규 ns**라 진짜 겪는다.

**대응(채택):** namespaces 변경을 **선행**시킨다 — 가능하면 Task 1(namespaces + test_psa)을 먼저 커밋·머지해 namespaces-prod가 homepage ns(restricted)를 만든 뒤 homepage-prod를 도입한다. 단일 PR로 가더라도 머지 후 **namespaces-prod Healthy + homepage ns 존재를 먼저 확인**한 뒤 homepage-prod 동기화를 관찰하고, 자동 수렴하지 않으면 명시 resync한다: `kubectl -n argocd patch app homepage-prod --type merge -p '{"operation":{"sync":{}}}'`. 이 recovery path는 Task 12에 명시한다.

**Step 1: 실패하는 테스트 작성**

Create `platform/namespaces/prod/test_homepage_ns.bats`:
```bash

@test "homepage namespace exists" {
  run grep -qE '^\s*name:\s*homepage\s*$' "$N"; [ "$status" -eq 0 ]
}

@test "homepage namespace enforces restricted PSA" {
  # homepage 블록 내 enforce=restricted 확인(yq로 정밀)
  run bash -c "yq e '.metadata.name == \"homepage\"' \"$N\" -N -o=json | grep -q true || \
               yq ea 'select(.metadata.name==\"homepage\").metadata.labels.\"pod-security.kubernetes.io/enforce\"' \"$N\" | grep -qx restricted"
  [ "$status" -eq 0 ]
}
```

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: homepage
  namespace: homepage
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: homepage
rules:
  # 서비스 자동발견(Gateway API HTTPRoute) — read-only
  - apiGroups: ["gateway.networking.k8s.io"]
    resources: ["httproutes", "gateways"]
    verbs: ["get", "list"]
  # pod-selector 상태 표시 + cluster 모드 발견 — read-only
  - apiGroups: [""]
    resources: ["namespaces", "pods", "nodes"]
    verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: homepage
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: homepage
subjects:
  - kind: ServiceAccount
    name: homepage
    namespace: homepage
```
